## 3. 总体产品结构

必牛软件不是单一 BI 报表，也不是单一任务工具，而是一个覆盖经营决策、任务执行和流程沉淀的闭环系统。

```text
BI 指标看板
  -> 规则识别
  -> 商品/店铺打标签
  -> 智能运营提示
  -> 工作台任务
  -> SOP 模板
  -> 执行与复盘
```

它对应文章中的三层框架：

| 文章框架 | 软件中的落地形态 | 对 1688 运营助手的启发 |
| --- | --- | --- |
| 决策流精准化 BI | 指标设置、趋势对比、类目均值、类目比值、商品/店铺排行 | 必须有趋势 BI、卡点排行、目标差距，而不是只给今日任务 |
| 业务流智动化 AI | 智能诊断、规则设置、自动执行、智能运营标签 | 规则引擎要把指标差距变成可执行动作 |
| 工作流标准化 SOP | 工作台、我的工作、项目管理、目标管理、SOP 管理 | 员工端应是今日事项 + SOP checklist + 次日验证 |

## 4. BI 页面观察

### 4.1 页面定位

主页面为商品排行/商品运营视角，支持按不同页面模板切换，例如：

- 全店动销
- 无级别
- E/D/C/B 等级
- 商品：爆款等级
- 商品：梯队详情
- 商品：销售梯队
- 商品：关联销售
- 品种/类目/价位/团队增长复盘
- 新品：预热、首日、首周、次周、详情分析
- 产品：布局
- 视觉：综合
- 供应：综合
- 运营：录数
- 财务：综合
- 管理：周常
- 会议：经营

这说明它的 BI 不是一个固定仪表盘，而是按运营问题拆出多个经营页面。

### 4.2 通用筛选维度

常见筛选字段包括：

- 统计周期：昨日、7日、15日、30日、日、周、月、年
- 店铺
- 商品
- 类目
- 负责人
- 链接上新
- 品种标签
- 备注标签
- 表现标签
- 爆款等级
- 上架状态
- 发货时间

对 1688 运营助手的启发：

```text
第一版可以不做复杂筛选器，
但数据结构必须保留店铺、商品、负责人、时间、目标、标签这些维度。
否则后续无法做管理者 BI 和卡点归因。
```

### 4.3 指标展示能力

页面支持：

- 同比
- 环比
- 自定义时间比
- 类目均值
- 类目比值
- 指标设置
- 更多设置
- 汇总行
- 表格排序

这说明它不只看当前值，而是强调对比和相对位置。

对 1688 运营助手的启发：

- 当前值只回答“现在怎么样”
- 趋势回答“是不是变好”
- 对标回答“比同类差在哪里”
- 目标差距回答“离平台/经营目标还差多少”

### 6.2 规则示例

观察到的示例规则：

```text
人群加购成本 > 7.5
且 人群花费 > 30
且 支付人数 = 0
-> 自动任务：待做提示
-> 标记：E
-> 完成后驻留 7 天
```

另一个示例：

```text
人群加购成本 > 0
且 人群加购成本 < 5
且 花费（含短视频） > 30

或 支付人数 > 0
且 成交成本 < 60

或 人群直接加购率 > 7%
且 人群花费 > 30

或 人群加购率 > 10%
且 人群花费 > 30

-> 自动任务：待做提示
-> 标记：D级
-> 完成后驻留 7 天
```

### 6.3 对我们的关键启发

规则引擎不是只输出“异常提醒”，而是完成四件事：

```text
指标条件命中
  -> 打标签
  -> 生成待办
  -> 绑定 SOP
```

对 1688 运营助手，应设计成：

| 规则来源 | 示例规则 | 触发动作 |
| --- | --- | --- |
| 官方规则 | 3 分钟响应率低于目标 | 生成客服值班/快捷回复检查/高峰时段排班任务 |
| 官方规则 | 找工厂履约率不足 | 生成交付排查/延期订单处理/现货承诺调整任务 |
| 官方规则 | 金冠品数量不足 | 生成潜力品筛选/标题主图优化/交易破零任务 |
| 内部经营 | 访客高但询盘低 | 生成详情页信任点/价格带/客服承接检查任务 |
| 内部经营 | 推广花费高但支付低 | 生成暂停推广/人群排查/关键词否词任务 |
| 内部经营 | 退款率升高 | 生成品质原因归类/详情页预期修正/供应链反馈任务 |

## 7. 工作台观察

工作台是覆盖在 BI 页面上的执行层，包含四个主模块：

```text
我的工作
项目管理
目标管理
SOP管理
```

这验证了一个重要判断：

```text
成熟的运营软件不会停在“看数据”，
而是把数据问题继续推到目标、项目、事项和 SOP。
```

### 7.1 目标管理

目标管理支持：

- 查看范围：我负责的、我关注的、全部
- 目标分类：周目标、月目标、年目标
- 目标周期
- 目标/KR 搜索
- 成员筛选
- 新建目标
- S/A/B/C 分层展示

对 1688 运营助手的启发：

- 目标要有层级，不是所有目标同权重
- 员工只看自己负责的目标
- 管理者看全部目标和目标分层
- 周/月/年目标要能互相承接

建议目标结构：

```text
年度经营目标
  -> 月度官方目标/经营目标
  -> 周度卡点目标
  -> 今日员工动作
```

### 7.2 项目管理

项目管理字段包括：

- 预计开始
- 预计结束
- 项目名称
- 进度
- 状态
- 项目负责人
- 项目成员
- 分组
- 目标
- 创建人
- 创建时间

项目状态按：

- 进行中
- 未开始
- 已完成

对 1688 运营助手的启发：

以下事项不适合只做成单日任务，更适合做成项目：

- 找工厂 L4/L5 冲刺
- 新灯塔服务分提升
- 旺旺 3 分钟响应专项
- 金冠品数量提升
- 新品首周破零
- 退款率压降
- 保温杯冬季货盘准备

### 7.3 我的工作

我的工作支持：

- 列表/月视图
- 新建事项
- 从 SOP 添加
- 批量设置
- 分组
- 更多筛选
- 表头设置
- 日期范围
- 成员
- 事项搜索
- 商品关联
- 查看范围
- 未完成
- 未安排
- 逾期
- 我关注的
- 我创建的
- @我的
- 预计工时
- 完成工时
- 剩余工时

对 1688 运营助手的启发：

员工端应该显示：

```text
今日事项
  -> 对应目标
  -> 对应商品/店铺
  -> 触发原因
  -> SOP checklist
  -> 完成勾选
  -> 次日/7日验证指标
```

不应让员工大量填写“动作证据”。更好的方式是：

- 默认 checklist 勾选
- 必要时才要求备注
- 高风险动作才要求截图或链接
- 系统自动记录执行时间和执行人
- 次日用指标变化判断动作是否有效

### 7.4 SOP 管理

SOP 管理分为：

- 个人 SOP
- 团队 SOP
- 系统 SOP

支持：

- 分组
- SOP 名称搜索
- 按创建时间排序
- 新建 SOP
- 分组管理
- 导入
- 编辑
- 查看
- 另存为
- 更多

对 1688 运营助手的启发：

SOP 应分三层：

| 层级 | 含义 | 示例 |
| --- | --- | --- |
| 系统 SOP | 软件内置，来自官方规则和通用运营方法 | 响应率补救 SOP、找工厂冲级 SOP、新品首周 SOP |
| 团队 SOP | 公司验证过的有效动作 | 保温杯询盘承接 SOP、杯盖漏水售后 SOP、礼品定制报价 SOP |
| 个人 SOP | 员工个人沉淀，可升级为团队 SOP | 某客服的话术、某运营的标题优化步骤 |

## 8. 团队 SOP「分级别运营」观察

这是本次探索中最关键的发现。

团队 SOP 中有一个模板：

```text
分级SOP / 分级别运营
```

表格字段包括：

- 序号
- 分类一
- 分类二
- 事项标题
- 说明
- 优先级
- 工时
- 处理人
- 自定义列
- 关联 SOP

### 8.1 模板内容

观察到的动作映射：

| 标记 | SOP 动作 |
| --- | --- |
| 无级别 | 加入低价引流计划 |
| E | 暂停推广 |
| D | 确保有 2 个买家秀；页面等级：D；加入品类动销计划 |
| C | 两天 5 单搜索单 + 买家秀；页面等级：C |
| B | 页面等级：B；点击率测图，目标点击率 > 5%；人群竞店计划；关键词相似品计划 |

### 8.2 底层机制

这不是普通 SOP 文档，而是：

```text
商品数据
  -> 系统标记等级
  -> 匹配等级 SOP
  -> 插入到个人工作
```

这对 1688 运营助手非常重要。我们的任务生成不应该是泛泛地“优化商品”，而应该是：

```text
商品/店铺/服务指标命中规则
  -> 获得标签
  -> 匹配 SOP 模板
  -> 生成今日 checklist
  -> 后续验证指标变化
```

### 8.3 对保温杯业务的转化

建议建立保温杯业务自己的分级 SOP：

| 标签 | 触发条件示例 | SOP 动作示例 |
| --- | --- | --- |
| 服务风险 | 3 分钟响应率低、满意度低 | 设置高峰值班；补齐快捷回复；检查 AI 客服是否有效；复盘未回复咨询 |
| 履约风险 | 48 小时揽收率低、履约率低 | 排查超时订单；调整现货承诺；暂停高风险定制接单 |
| 找工厂缺口 | 定制交易积分不足、合约支付率不足 | 推合约订单；整理定制案例；优化找工厂询盘承接 |
| 新品首日 | 上新 1 天内访客低或询盘低 | 检查标题关键词；补主图场景；设置基础价格带；引导首批询盘 |
| 潜力品 | 访客有增长但支付弱 | 优化详情页信任点；补买家秀；检查起订量和阶梯价 |
| 金冠候选 | 转化和复购较好但流量不足 | 加大站内曝光；优化主图点击率；绑定重点活动 |
| 退款风险 | 品质退款或物流退款升高 | 标注问题 SKU；回传供应链；修改详情页预期；客服提前解释 |

## 9. 商品梯队和生命周期观察

### 9.1 商品梯队详情

商品梯队页面使用月销台阶：

- 月销 300
- 月销 500
- 月销 1000
- 月销 2000
- 月销 3000
- 月销 4000

对我们的启发：

保温杯商品不应只分“好/坏”，而应按成长台阶管理：

```text
新品
  -> 破零款
  -> 潜力款
  -> 稳定款
  -> 金冠候选
  -> 主推爆款
  -> 利润款/复购款
```

### 9.2 新品阶段

软件把新品拆成：

- 新品：预热
- 新品：首日
- 新品：首周
- 新品：次周
- 新品：详情分析

对我们的启发：

新品不能套用成熟品规则。应按阶段设置不同判断重点：

| 阶段 | 重点指标 | 动作重点 |
| --- | --- | --- |
| 预热 | 标题、主图、价格、基础素材 | 上架前准备、关键词、定价、素材完整度 |
| 首日 | 曝光、访客、点击、询盘 | 检查点击率、价格带、基础承接 |
| 首周 | 加购、询盘、支付、客服反馈 | 判断是否有需求、是否要继续给流量 |
| 次周 | 转化、退款、复购、推广效率 | 决定进入潜力池、调整定位或止损 |

### 9.3 产品布局

产品布局下有几个维度：

- A时间：上新批次
- A时间：类目热图
- C宽度：类目分布
- B宽度：品种分布
- A宽度：价位分布
- A密度：爆款裂变
- S高度：销量梯队
- C高度：爆款优化
- C长度：延长季节

这说明软件把产品结构拆成：

```text
时间
宽度
密度
高度
长度
```

对保温杯业务的启发：

- 时间：季节上新节奏，秋冬保温杯旺季提前布局
- 宽度：儿童杯、商务杯、礼品杯、户外杯、咖啡杯等货盘宽度
- 价位：低价引流、中价成交、高价礼品/定制
- 密度：一个爆款能否裂变颜色、容量、材质、套装
- 高度：商品是否能从破零到金冠/主推
- 长度：淡季如何用礼品、定制、团购延长生命周期

## 12. 对 1688 运营助手的产品结论

### 12.1 当前产品必须升级的方向

这次探索后，1688 运营助手不应再只做“数据录入 + 今日建议”。更准确的产品形态应是：

```text
官方目标驱动的运营闭环系统
```

核心链路：

```text
规则版本
  -> 指标录入
  -> 目标差距
  -> 标签识别
  -> 项目/事项生成
  -> SOP checklist
  -> 动作回测
  -> SOP 进化
```

### 12.2 应优先补齐的模块

| 优先级 | 模块 | 为什么重要 |
| --- | --- | --- |
| P0 | 规则版本库 | 官方政策会更新，不能把旧规则当永久真理 |
| P0 | 趋势 BI | 员工和老板都要看到目标趋势和卡点 |
| P0 | 规则触发任务 | 数据必须能变成行动，否则只是报表 |
| P0 | SOP checklist | 降低员工执行摩擦，避免写长证据 |
| P1 | 项目管理 | 冲级和专项改善不是一天完成，需要项目化 |
| P1 | 动作回测 | 判断任务是否真的有效 |
| P1 | SOP 生命周期 | 把有效动作沉淀，把无效动作淘汰 |
| P2 | 自动执行 | 必须在规则稳定、人工确认充分后再做 |

### 12.3 员工端建议形态

员工打开软件后应看到：

```text
今日指挥台
  1. 今天最重要的目标差距
  2. 被规则命中的商品/服务/店铺指标
  3. 系统推荐的 SOP checklist
  4. 完成按钮和轻备注
  5. 明天要验证的指标
```

不建议员工端一开始展示完整 BI。完整 BI 应主要给主管/老板使用。

### 12.4 管理端建议形态

管理端应看到：

```text
趋势 BI
卡点排行
目标分层
项目进度
员工事项
SOP 有效率
规则版本状态
```

管理者关心的不是员工有没有填表，而是：

- 哪个目标最危险
- 哪个指标拖后腿
- 哪个负责人卡住
- 哪个 SOP 有效
- 哪条规则可能过期
- 下周资源该投哪里

## 13. 不应照搬的地方

### 13.1 不照搬复杂指标池

必牛指标非常多，但我们第一版不应复制大而全指标设置。原因：

- 员工会看不懂
- 数据录入成本太高
- 保温杯业务不需要所有淘系广告指标
- 1688 官方目标和找工厂目标更优先

### 13.2 不提前做自动执行

页面中有“自动执行”，但我们的系统第一阶段不应自动执行平台动作。

原因：

- 官方规则可能更新
- 店铺经营策略需要老板确认
- 自动暂停推广、调整商品等动作有风险
- 目前数据源多为人工录入，不能完全信任

第一阶段应做：

```text
自动诊断
人工确认
半自动任务
人工执行
自动回测
```

### 13.3 不让员工写重证据

工作台更像 checklist 和事项管理，而不是证据收集系统。

因此我们的执行记录应轻量：

- 默认勾选
- 可选备注
- 特定动作才要截图/链接
- 系统自动保存执行时间、负责人、关联指标

### 14.2 规则到任务的生成逻辑

  -> 查找 SOP 模板

下一版不需要一次做完整系统，最小闭环可以是：

```text

3. 生成 1 组 checklist
4. 员工勾选完成
5. 明天输入同一指标
6. 系统判断动作是否有效
7. 有效则进入 SOP 候选
```

### 14.4 保温杯业务专属 SOP 优先级

第一批 SOP 建议只做 6 类：

1. 旺旺 3 分钟响应率补救 SOP
2. 找工厂履约/合约支付补救 SOP
3. 新品首日检查 SOP
4. 潜力品转金冠 SOP
5. 退款率异常排查 SOP
6. 保温杯定制询盘承接 SOP

## 15. 关键结论

这次探索证明，真正有用的运营软件不是：

```text
报表软件
待办软件
SOP 文档库
AI 建议生成器
```

而是把四者串起来：

```text
数据看见问题
规则定义判断
SOP 提供动作
任务推动执行
回测验证结果
```

因此，1688 保温杯运营助手的下一阶段产品目标应调整为：

```text
让员工每天少判断、多执行；
让主管每周看趋势、抓卡点；
让老板每月看目标、看规则、看 SOP 是否真的有效。
```
