# 09 重构原则：从功能堆叠到经营闭环

本文档记录 1688 运营助手后续重构的产品原则。它不是立即开发计划，而是判断功能取舍、页面重构和架构演进时的底层标准。

## 一句话定位

1688 运营助手不是数据看板，也不是任务管理器。

它应该是一个基于规则、指标和保温杯类目知识的运营闭环系统：

```text
监测 -> 分析 -> 行动 -> 复盘
```

对应底层链路：

```text
目标 -> 指标 -> 数据 -> 卡点 -> 动作 -> 验证 -> SOP
```

## 第一原则

```text
好的运营，是把一个模糊目标拆到刚好能执行、能归因、能验证的颗粒度。
```

运营问题首先是颗粒度问题，其次是因果链问题和验证周期问题。

### 颗粒度太粗

```text
提升转化
优化商品
提高回复率
做好复盘
```

员工不知道今天具体做什么。

### 颗粒度太细

```text
改第 3 个商品第 2 张图
写第 7 条话术
检查某个客户
```

老板看不出它和经营目标有什么关系。

### 合适颗粒度

```text
补齐 09:00-21:00 客服值班，并复盘 20 条未响应消息原因。
```

因为它满足：

- 目标能追溯：对应响应率、服务体验、找工厂牌级。
- 员工能执行：今天能完成。
- 明天能验证：看 3 分钟响应率是否提升。
- 复盘能沉淀：能判断是排班、提醒、话术还是低质量询盘。

## 重构方向

现有页面不应继续无限增加标签页。长期体验应收敛成四个核心模块：

| 模块 | 目的 | 核心问题 |
| --- | --- | --- |
| 监测 | 看当前阶段最关键的少数指标。 | 现在哪里异常？数据新不新？ |
| 分析 | 把指标异常翻译成卡点。 | 为什么异常？卡在哪里？ |
| 行动 | 生成今日作战单。 | 今天谁做什么，做到什么程度？ |
| 复盘 | 用数据判断动作效果。 | 动作有效吗，保留、调整还是停止？ |

## 指标原则

不要一开始展示所有指标。员工端只展示当前阶段最关键的少数指标。

指标分层：

| 层级 | 指标 |
| --- | --- |
| 服务底线 | 3 分钟响应率、退款处理时长、品质退款率、48 小时揽收率。 |
| 流量承接 | 曝光、访客、访客询盘率、广告曝光占比。 |
| 成交闭环 | 询盘数、询盘转支付、合约支付率、支付金额。 |
| 平台目标 | 找工厂牌级、L 等级、潜力商品、采购指数4.0商品、镇店之宝。 |
| 经营安全 | 毛利率、广告费率、履约风险、售后风险。 |
| 能力沉淀 | 有效动作数、停止动作数、SOP 数。 |

原则：

```text
不是固定少数指标，而是当前阶段只暴露少数指标。
```

## 行为设计原则

可以借鉴行为心理学，但不应做幼稚游戏化。

不优先做：

- 积分排行榜。
- 夸张奖励动画。
- 为完成任务而完成任务的打卡系统。

优先做：

- 少量任务。
- 低摩擦勾选。
- 即时反馈。
- 明日验证。
- 自动复盘。
- 有效动作进入 SOP。

核心不是奖励员工“做了”，而是让员工看到“动作让指标动了”。

## 数据采集原则

API 是长期方向，但不能成为当前产品验证的前置条件。

数据采集路径按顺序演进：

```text
手填关键指标
XLS 上传
复制表格
截图 OCR
官方 API / ERP API
```

只要能支持今日作战单和明日验证，就可以先跑起来。

## 规则引擎原则

不要一开始做通用复杂规则引擎。先做配置化知识库：

```text
指标字典
数据源映射
目标-指标映射
指标-卡点映射
卡点-动作映射
动作-验证映射
规则版本库
保温杯类目知识
```

这些稳定后，再抽象成规则引擎。

## 技术架构原则

当前阶段不建议微服务。问题还不是系统规模，而是经营模型仍在成型。

建议采用模块化单体：

| 模块 | 责任 |
| --- | --- |
| 数据导入模块 | 处理 XLS、手填、截图、未来 API。 |
| 指标计算模块 | 计算趋势、差距、漏斗和状态。 |
| 规则映射模块 | 管理目标、指标、规则、卡点和动作映射。 |
| 作战单模块 | 生成每日主目标和 1-3 个动作。 |
| 复盘模块 | 判断动作效果，沉淀 SOP。 |

只有当多店铺、多员工、通知、API 同步变复杂时，再考虑服务拆分。

## 版本验证原则

每个版本发布后，不只看功能是否完成，还要看产品是否真的让员工变好。

应跟踪：

| 指标 | 含义 |
| --- | --- |
| 使用率 | 员工是否每天打开。 |
| 作战单完成率 | 动作是否可执行。 |
| 数据导入完成率 | 系统是否有足够判断依据。 |
| 问题解决率 | 最大卡点是否下降。 |
| 平均操作时间 | 是否低摩擦。 |
| 有效动作率 | 完成动作后指标是否改善。 |
| SOP 沉淀数 | 是否形成长期资产。 |

## 功能取舍标准

一个功能是否值得做，先问：

1. 它是否服务“监测、分析、行动、复盘”之一。
2. 它是否让目标、数据、卡点、动作、验证关系更清楚。
3. 它是否降低员工执行摩擦。
4. 它是否能产生可复盘的数据。
5. 它是否能沉淀为长期基座，而不是一次性页面。

如果都不能，就不要做。
