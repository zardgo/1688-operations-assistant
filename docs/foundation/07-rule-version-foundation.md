# 07 规则版本基座

规则版本基座回答一个问题：系统怎样引用官方规则，而不把旧政策当永久真理。

1688 的官方规则会更新，店铺后台也可能按类目、层级、业务类型展示不同口径。因此软件不能只写死“找工厂铜牌要求多少”，而要记录规则来源、发布日期、适用范围、来源可信度和替代关系。

## 规则对象

| 字段 | 说明 |
| --- | --- |
| `rule_id` | 稳定唯一 ID。 |
| `rule_name` | 规则名称。 |
| `rule_type` | `official` 官方规则，`internal` 内部经营规则，`sample` 样本经验。 |
| `source_url` | 官方链接或后台截图来源。 |
| `published_at` | 发布日期。 |
| `collected_at` | 收集日期。 |
| `effective_from` | 生效日期。 |
| `effective_to` | 失效日期，没有则为空。 |
| `scope` | 适用范围，例如找工厂、消费品、保温杯、店铺等级。 |
| `metric_ids` | 影响哪些指标。 |
| `goal_ids` | 影响哪些目标。 |
| `status` | `draft`、`source_found`、`active`、`deprecated`。 |
| `source_confidence` | 来源可信度：`low`、`medium`、`high`。 |
| `last_reviewed_at` | 最近维护日期。 |
| `review_notes` | 来源、口径或后台覆盖说明。 |
| `supersedes_rule_id` | 替代的旧规则。 |
| `notes` | 口径备注。 |

## 规则状态

| 状态 | 含义 | 软件使用方式 |
| --- | --- | --- |
| `draft` | 来自假设、旧资料或未核对内容。 | 只能做内部讨论和弱提示。 |
| `source_found` | 找到官方来源、后台截图、用户摘录或飞书资料。 | 可以生成动作建议，但必须展示来源和版本。 |
| `active` | 当前软件采用的规则版本。 | 可以驱动目标差距、提醒和 checklist。 |
| `deprecated` | 已被新规则替代或不再适用。 | 只能用于历史解释。 |

核心原则：

```text
不再用人工确认卡住员工端使用。
规则必须带版本、来源和可信度；来源不足时显示“来源待补”，但不阻塞行动建议。
后台当前口径可以覆盖规则库默认口径。
```

## 规则和目标的关系

规则不是目标本身，而是目标的判断依据。

| 目标 | 可能引用规则 |
| --- | --- |
| 找工厂铜牌/银牌/金牌 | 找工厂牌级规则、找工厂服务响应规则、找工厂履约规则。 |
| 服务体验修复 | 商家服务管理规范、咨询体验规则、售后体验规则。 |
| 店铺 L 等级成长 | 消费品行业店铺等级评价规则、商品成长规则。 |
| 商品成长优化 | 采购指数4.0规则、新品采购指数规则、镇店之宝规则。 |
| 交易增长 | 平台交易、营销、广告相关规则。 |

目标可以同时引用多个规则。规则变化时，不改目标定义，而是新增规则版本并调整映射。

## 规则和指标的关系

规则要落到指标上，否则无法被系统执行。

示例：

```text
规则：找工厂牌级规则 2026-05
目标：冲找工厂铜牌
指标：
- 找工厂服务响应率 >= 60%
- 找工厂履约率 >= 70%
- 月动销小单定制商品数 >= 3
- 定制交易积分 >= 100000
- 合约支付率 >= 50%
```

如果这些阈值缺来源或和后台不一致，系统应显示“来源待补/以后台为准”，但仍可按当前录入目标生成 checklist。

## 规则冲突处理

可能出现三类冲突：

| 冲突类型 | 处理原则 |
| --- | --- |
| 官方规则之间冲突 | 以发布日期更新、适用范围更精确、后台当前展示为优先。 |
| 官方目标和经营安全冲突 | 经营安全优先，例如毛利、履约、售后风险。 |
| 内部经验和官方规则冲突 | 官方规则用于平台目标，内部经验用于经营建议，不能互相冒充。 |

例子：

```text
为了冲定制交易积分继续接低毛利定制单。
```

系统应阻断，因为毛利和履约是经营底线。

## 规则更新流程

1. 新发现规则时，新增 `draft` 规则卡。
2. 填写来源、发布日期、适用范围和影响指标。
3. 补充 `source_confidence`、最近维护时间和后台覆盖说明。
4. 当前采用版本状态改为 `active`。
5. 如果替代旧规则，旧规则改为 `deprecated`，但不删除。
6. 软件生成作战单时，优先使用当前目标下的 `active` 规则；`source_found` 可生成弱提示和 checklist。

## 第一批需要维护的规则域

| 规则域 | 重要性 | 当前处理 |
| --- | --- | --- |
| 找工厂牌级规则 | 高 | 作为高优先级规则版本维护，来源不足时显示来源待补。 |
| 商家服务管理规则 | 高 | 影响响应、售后、退款、介入。 |
| 消费品店铺等级规则 | 中 | 影响 L 等级和成长目标。 |
| 商品成长规则 | 中 | 影响潜力商品、采购指数4.0商品、货盘建设。 |
| 采购指数4.0规则 | 高 | 2026-05-25 后替代金冠品体系，影响商品扶持、镇店之宝和商品成长目标。 |
| 广告/推广规则 | 中 | 影响投放动作边界。 |
| 售后/争议规则 | 高 | 影响品质退款、平台介入、服务分。 |

## 规则基座的长期价值

规则版本基座不是为了把政策文档存起来，而是为了让系统知道：

```text
这条建议依据什么规则
这条规则是否仍然有效
这条规则适用于哪个目标
这条规则能不能驱动员工动作
```
