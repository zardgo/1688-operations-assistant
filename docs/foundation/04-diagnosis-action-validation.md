# 04 诊断-动作-验证闭环

这个文档定义系统如何从数据异常走到员工行动，再从结果回到 SOP。

软件如果只有 Checklist，员工会变成填表。软件如果有诊断和验证，员工才会逐渐形成经营判断。

## 闭环结构

```text
指标异常
  -> 卡点诊断
  -> 原因假设
  -> 今日动作
  -> 明日/周期验证
  -> 复盘判断
  -> SOP 或停止
```

## 诊断层

诊断层不直接派任务，而是先判断“问题属于哪一类”。

| 诊断字段 | 说明 |
| --- | --- |
| `symptom` | 表面现象，例如“3 分钟响应率低”。 |
| `bottleneck_type` | 卡点类型，例如“接待机制卡点”。 |
| `hypothesis` | 原因假设，例如“午休和晚间无人兜底”。 |
| `evidence_to_check` | 需要验证的证据，例如“未响应消息按时间段分布”。 |
| `confidence` | 高、中、低。 |
| `blocked_actions` | 当前不建议做什么，例如“先不加广告”。 |

## 动作层

动作必须是员工今天能做的，不是抽象建议。

| 动作字段 | 说明 |
| --- | --- |
| `action_id` | 动作模板 ID。 |
| `title` | 动作标题。 |
| `owner_role` | 客服员工、运营员工、负责人。 |
| `target_metric` | 主要影响指标。 |
| `steps` | 具体执行步骤。 |
| `due_time` | 今日截止时间。 |
| `completion_mode` | 默认是 checkbox，不强制证据。 |
| `optional_note` | 可选备注，用于异常说明。 |
| `evidence_trigger` | 只有高风险或复盘需要时才要求证据。 |

## 验证层

每个动作生成时，就要同时生成验证方式。

| 验证字段 | 说明 |
| --- | --- |
| `verification_metric` | 用什么指标判断。 |
| `baseline_value` | 动作前基线。 |
| `expected_direction` | 上升、下降、稳定。 |
| `expected_value` | 目标值。 |
| `verification_window` | 明天、3 天、7 天、30 天。 |
| `success_criteria` | 什么算有效。 |
| `stop_condition` | 什么情况下停止动作。 |

## 复盘判断

复盘不要让员工写长作文，先结构化。

| 结果 | 判断 | 下一步 |
| --- | --- | --- |
| `effective` | 指标明显改善。 | 保留动作，进入 SOP 候选。 |
| `watch` | 有改善但不足。 | 再跑一个周期，或补充原因假设。 |
| `ineffective` | 没改善或反向。 | 停止动作，换假设。 |
| `unclear` | 数据缺失或样本太小。 | 先补数据，不评价员工。 |

## SOP 晋级条件

动作不能一次有效就永久 SOP。建议条件：

1. 同一动作至少通过 2 次复盘。
2. 对应指标改善不是偶然样本波动。
3. 员工能写清楚适用场景和不适用场景。
4. 新员工照做不会产生明显误解。
5. 负责人确认不会伤害毛利、履约或售后。

## 示例：响应率低

```text
指标异常：
旺旺 3 分钟响应率 52%，目标 60%。

卡点诊断：
接待机制卡点。

原因假设：
午休、晚间、客服离开时没有兜底；保温杯高频问题没有快捷回复。

今日动作：
1. 设置 09:00-21:00 值班兜底。
2. 整理 20 条保温杯高频快捷回复。
3. 复盘 20 条未响应消息，按时间段和问题类型归因。

验证：
明天看 3 分钟响应率是否从 52% 回升到 60% 以上。

停止条件：
连续 3 天无改善，停止继续写话术，改查排班或低质量询盘占用。
```

## 低摩擦原则

1. 员工端默认只需要勾选完成。
2. 备注可选，只记录异常原因。
3. 证据不常态化要求，避免员工为了证明而工作。
4. 系统用第二天数据自动验证动作效果。
5. 管理者看动作是否带动指标，不只看完成率。
