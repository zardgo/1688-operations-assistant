# 06 基座治理与维护

这些底层表以后会变成软件内置知识库，但现在先用 Markdown 管理。关键不是一次写全，而是保证后续能更新、能追溯、能废弃。

## 文档状态

建议每个底层条目都带状态：

| 状态 | 含义 |
| --- | --- |
| `draft` | 初稿，来自经营判断或样本，只做弱提示。 |
| `source_found` | 找到来源，可生成建议动作，但要显示来源和可信度。 |
| `active` | 当前软件采用版本，可进入目标差距和 checklist。 |
| `validated` | 已通过真实执行和复盘验证。 |
| `deprecated` | 已过期或被更好版本替代。 |

## 官方规则治理

官方规则必须单独治理：

| 字段 | 要求 |
| --- | --- |
| 规则名称 | 必填。 |
| 发布日期 | 必填，不能只写“最新”。 |
| 官方链接 | 必填，或者记录后台截图来源。 |
| 适用范围 | 必填，例如找工厂、消费品、保温杯、店铺等级。 |
| 指标口径 | 必填，尽量保留原始口径。 |
| 来源可信度 | 必填，`low`、`medium` 或 `high`。 |
| 最近维护时间 | 记录最近一次补来源、改口径或后台覆盖的时间。 |
| 替代关系 | 如果新规则覆盖旧规则，要保留旧规则但标记废弃。 |

原则：

```text
官方规则可以预置，不用人工确认卡住员工端使用。
来源不足时显示“来源待补”，后台当前口径可以覆盖规则库默认口径。
```

## 动作库治理

动作不能越积越多。每条动作都要经过复盘。

| 动作状态 | 含义 |
| --- | --- |
| `candidate` | 候选动作，来自经验或假设。 |
| `active` | 当前可派发动作。 |
| `validated` | 多次验证有效，可进入 SOP。 |
| `limited` | 只在特定场景适用。 |
| `stopped` | 已验证无效或副作用大。 |

动作必须有：

- 触发卡点。
- 负责人角色。
- 今日执行步骤。
- 验证指标。
- 验证周期。
- 停止条件。
- 不适用场景。

## 指标治理

指标最容易乱，所以要控制口径。

1. 同一个指标只能有一个稳定 ID。
2. 后台字段名变化时，不改指标 ID，只更新数据源映射。
3. 口径不确定的指标标记为 `draft`。
4. 数据源缺失时，系统不能假装知道。
5. 指标目标值必须说明来源：官方规则、内部经验、样本基准或临时目标。

## 复盘治理

复盘的目的不是考核员工，而是更新知识库。

复盘要回答：

1. 动作做了吗。
2. 指标动了吗。
3. 如果动了，是否可能是该动作造成。
4. 如果没动，是执行不足、假设错误、数据缺失，还是周期太短。
5. 这个动作应保留、重复、调整、停止，还是进入 SOP 候选。

## 从 Markdown 到软件内置的门槛

一条内容进入软件内置前，至少满足：

- 字段结构清楚。
- 来源或假设清楚。
- 适用范围清楚。
- 不适用场景清楚。
- 有验证方式。
- 负责人确认。

优先内置顺序：

```text
指标字典
数据源映射
目标-指标映射
指标-卡点映射
卡点-动作映射
动作-验证映射
保温杯类目知识
规则版本库
```

## 维护节奏

| 节奏 | 动作 |
| --- | --- |
| 每日 | 只记录数据缺口、动作执行和验证结果。 |
| 每周 | 更新动作状态，沉淀 SOP 候选，停止无效动作。 |
| 每月 | 检查指标口径、目标映射、类目知识是否需要更新。 |
| 官方规则更新时 | 新增规则版本，标记 `source_confidence`，采用后设为 `active`。 |
