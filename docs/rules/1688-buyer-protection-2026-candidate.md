## 1. 可借鉴的核心判断

买家保障不应只看作“店铺装饰服务”，而应纳入服务履约模块。它和新灯塔、商品质量、活动准入、买家信任有关。

对保温杯店铺有价值的抽象是：

- 保障服务要和真实履约能力匹配，不能为了图标乱开。
- 保障应拆成两类：履约类保障、售后类保障。
- 员工动作应该是 checklist：哪些商品缺保障、哪些保障开了但履约风险高、哪些服务承诺和库存/发货能力不一致。

## 2. 不应直接写死的说法

下列说法来自 AI 摘录，当前不能当作官方规则：

| 说法 | 处理方式 |
| --- | --- |
| 未达标商品会被直接屏蔽搜索与活动入口 | 需要官方规则或后台提示验证 |
| 保温杯/厨房用品属于 2026 Q2 强制保障类目 TOP10 | 需要类目后台或规则链接验证 |
| 必须包含 1 项履约类 + 1 项售后类 | 可作为 checklist 假设，不作为硬门槛 |
| 开通 3 项以上且履约达标自动激活 7 星流量 | 暂不进入员工端承诺文案 |
| 三无包赔、定制专属保障会触发具体流量通道 | 需要后台实际权益页验证 |

## 3. 软件字段建议

| 字段 | 含义 | 来源 | 动作方向 |
| --- | --- | --- | --- |
| `buyer_protection_enabled_count` | 商品已开通保障服务数量 | 商品编辑页/服务保障页 | 识别缺服务商品 |
| `fulfillment_protection_enabled` | 是否有履约类保障 | 商品编辑页/服务保障页 | 补发货、运踪、交付承诺类服务 |
| `after_sales_protection_enabled` | 是否有售后类保障 | 商品编辑页/服务保障页 | 补破损、少货、退换类服务 |
| `custom_protection_enabled` | 定制商品是否有定制相关保障 | 哇噢定制/商品编辑页 | 支撑定制流量承接 |
| `protection_breach_rate` | 保障承诺违约率 | 售后/物流/新灯塔 | 防止“开通即失信” |
| `shipping_promise_hours` | 发货承诺时效 | 商品编辑页/物流服务 | 对齐库存和仓配能力 |

## 3.1 动作频率

买家保障属于典型的一次性配置动作。开通前可以作为 checklist；开通后应转为监控项。

| 状态 | 系统动作 |
| --- | --- |
| 未开通 | 生成一次性开通任务 |
| 已开通 | 不再生成日常开通任务 |
| 新增商品 | 检查新商品是否继承或补齐保障 |
| 保障承诺违约 | 生成异常处理任务，排查库存、发货、售后承诺 |
| 月度复核 | 检查保障是否仍匹配真实履约能力和平台新规则 |

## 4. 员工 checklist

| 场景 | checklist |
| --- | --- |
| 主推现货商品 | 是否至少有一个履约类保障；是否和真实库存、发货时效一致 |
| 易碎/易损保温杯 | 是否有破损、少货、售后类保障；包装 SOP 是否同步 |
| 定制商品 | 是否有定制相关保障；确认稿、交期、售后边界是否写清 |
| 已开保障但投诉增加 | 先排查承诺是否过度，再看客服和售后处理时长 |

## 5. 规则关系

| 关联模块 | 关系 |
| --- | --- |
| 新灯塔 | 买家保障会影响服务体验、售后体验和履约风险。 |
| 商品质量星级 | 保障完整度可能影响发布质量和买家信任，但具体星级机制待补来源。 |
| 采购指数4.0 | 服务、品质、商誉维度可能受保障履约影响。 |
| L 等级 | 服务健康度差会削弱冲级确定性。 |
