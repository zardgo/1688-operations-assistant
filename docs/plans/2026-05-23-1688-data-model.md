# 1688 官方目标迭代驾驶舱数据模型拆解

## 1. 设计原则

数据模型服务一个目标：让每个员工动作都能追溯到“官方目标、经营底线、指标差距、执行结果、SOP 沉淀”。

因此第一版不以页面为中心建模，而以运营因果链建模：

```text
RuleSource
  -> MetricDefinition
  -> GoalPreset
  -> IterationCycle
  -> MetricReading
  -> DiagnosticGap
  -> TaskInstance
  -> ExecutionLog
  -> WeeklyReview
  -> SopEntry
```

核心原则：

- 官方规则和经营判断分开存。
- 指标定义和指标读数分开存。
- 目标值和当前值分开存。
- 任务模板和员工当天任务分开存。
- 每个任务必须绑定指标或经营底线。
- 每个指标必须有数据口径、统计周期、适用范围和置信度。
- 百分比指标必须保留分子、分母和剔除口径，不能只存最终百分比。
- 每周复盘必须保留当时的指标快照，不能只存指标 ID。
- 适用范围必须机器可判断，不能只写自然语言备注。

## 2. 核心实体

### 2.1 `RuleSource`

官方规则来源。

字段：

- `id`
- `name`
- `url`
- `publishedAt`
- `effectiveAt`
- `sourceType`: `official_rule | official_notice | internal_method`
- `applicability`: 适用范围，例如“找工厂商家，不含超级工厂”
- `status`: `pending_confirmation | confirmed | possibly_outdated`
- `lastReviewedAt`
- `notes`

用途：

- 防止旧规则被当作永久真理。
- 每个官方指标都能回溯到规则来源。

### 2.2 `MetricDefinition`

指标定义，不存具体值。

字段：

- `id`
- `name`
- `priority`: `P0 | P1 | P2 | P3`
- `category`: `platform_survival | factory_level | l_growth | business_iteration | guardrail`
- `direction`: `min | max | range`
- `unit`: `percent | count | score | yuan | day | hour`
- `period`: 例如 `daily`, `last_30_days`, `T-33_to_T-4`
- `formula`
- `dataSourceHint`: 员工从 1688 后台哪里抄
- `applicability`
- `applicabilityRules`: 机器可判断的适用条件
- `exclusionRules`: 官方剔除逻辑，例如合约/定制/邀约订单是否剔除
- `sourceInstruction`: 后台路径、截图位置或手工录入说明
- `valueInterpretation`: 数值越高越好、越低越好或区间合格
- `ruleSourceId`
- `isOfficial`

示例：

```json
{
  "id": "ww_3min_response_rate",
  "name": "近30天旺旺3分钟响应率",
  "priority": "P0",
  "category": "platform_survival",
  "direction": "min",
  "unit": "percent",
  "period": "last_30_days",
  "formula": "3分钟内有效回复买家咨询消息数 / 买家咨询消息数",
  "dataSourceHint": "1688后台服务/新灯塔/咨询体验",
  "applicabilityRules": {
    "categoryType": ["consumer_goods", "mixed"],
    "orderType": ["spot", "custom", "contract", "invite", "mixed"]
  },
  "exclusionRules": [
    "店铺主账号与子账号、子账号之间消息不计入",
    "官方小二账号消息不计入",
    "退款卡片不计入",
    "普通旺旺/千牛自动回复不算有效回复"
  ],
  "isOfficial": true
}
```

### 2.3 `GoalPreset`

员工当前选择的阶段目标。

字段：

- `id`
- `name`
- `goalType`: `protect_service | factory_bronze | factory_silver | factory_gold | l_growth | lighthouse_repair`
- `description`
- `requirements`: `GoalRequirement[]`
- `blockedBy`: 经营底线或入场资格限制
- `recommendedReviewCadence`: `daily | weekly`
- `entryRequirements`: 目标入场资格，例如是否已入驻找工厂
- `guardrailPolicy`: 冲目标时必须遵守的经营底线

找工厂目标示例：

```json
{
  "id": "factory_bronze",
  "name": "冲找工厂铜牌",
  "requirements": [
    { "metricId": "service_response_rate_30d", "target": 0.6 },
    { "metricId": "fulfillment_rate_30d", "target": 0.7 },
    { "metricId": "custom_trade_points_30d", "target": 100000 },
    { "metricId": "contract_payment_rate_30d", "target": 0.5 },
    { "metricId": "monthly_active_small_custom_sku_count", "target": 3 }
  ]
}
```

### 2.4 `ShopContext`

店铺上下文，决定哪些规则适用。

当前项目默认值：

- `primaryCategory`: `保温杯`
- `categoryType`: `consumer_goods`
- `mainOrderType`: `mixed`

字段：

- `shopName`
- `primaryCategory`
- `categoryType`: `consumer_goods | mixed | other | unknown`
- `mainOrderType`: `spot | custom | contract | invite | mixed`
- `isFactoryChannelJoined`
- `isSuperFactory`
- `isCustomFactory`
- `currentFactoryLevel`: `none | bronze | silver | gold`
- `currentLLevel`
- `grossMarginFloor`
- `ownerDailyInvolvementLimitHours`
- `defaultGoalId`
- `dataReviewCadence`
- `manualEntryOwner`

用途：

- 对保温杯消费品，P0 物流体验优先看 48 小时揽收率；找工厂牌级仍看牌级规则里的履约率。
- 判断是否有资格冲找工厂牌级。
- 判断目标是否被经营底线阻断。

### 2.5 `MetricReading`

某个时间点的指标读数。

字段：

- `id`
- `metricId`
- `value`
- `numerator`
- `denominator`
- `excludedCount`
- `targetValue`
- `recordedAt`
- `periodStart`
- `periodEnd`
- `source`: `manual | import | calculated`
- `confidence`: `low | medium | high`
- `confirmedByUser`
- `sourceLabel`: 后台展示出来的字段名
- `rawInput`: 员工原始录入值，保留给审计和纠错
- `notes`

用途：

- 允许员工手动录入。
- 保留历史，支持周复盘判断进步/退步。
- 通过分子、分母和剔除数量解释“百分比变化是真的变好，还是样本量变化造成”。

### 2.5.1 `IterationCycle`

一次每日执行或每周复盘周期。

字段：

- `id`
- `type`: `daily | weekly`
- `startedAt`
- `endedAt`
- `selectedGoalId`
- `shopContextSnapshot`
- `metricReadingIds`
- `generatedTaskIds`
- `reviewId`
- `status`: `open | completed | skipped`

用途：

- 把目标、读数、任务和复盘绑在同一个周期里。
- 避免后续目标切换后，旧任务不知道属于哪个目标。
- 支持“每日执行 + 每周复盘”的产品节奏。

### 2.6 `DiagnosticGap`

系统根据目标和读数生成的差距。

字段：

- `metricId`
- `currentValue`
- `targetValue`
- `gapValue`
- `riskLevel`: `safe | watch | risky | critical`
- `priority`
- `impact`: 影响什么权益
- `diagnosis`: 为什么重要
- `blockedActions`: 当前不建议做什么

示例：

```json
{
  "metricId": "ww_3min_response_rate",
  "currentValue": 0.52,
  "targetValue": 0.6,
  "riskLevel": "risky",
  "impact": "影响咨询体验、商家等级、搜索推荐、活动报名",
  "diagnosis": "低于冲铜牌服务响应底线，应优先修客服响应，而不是扩大推广。"
}
```

### 2.7 `DecisionGuardrail`

目标宪法里的裁判规则。

字段：

- `id`
- `name`
- `condition`
- `severity`: `block | warn | inform`
- `message`
- `recommendedAction`

示例：

- 毛利低于底线时，阻止系统推荐冲 GMV。
- 48 小时揽收率或找工厂履约率低于目标时，阻止推广复杂定制单。
- P0 服务指标低于安全线时，暂停 P2/P3 扩张动作。

### 2.8 `TaskTemplate`

系统任务模板。

字段：

- `id`
- `triggerMetricId`
- `triggerCondition`
- `title`
- `why`
- `steps`
- `methodCardId`
- `expectedEvidence`
- `sopSuggestion`
- `unsafeWhen`: 哪些 guardrail 命中时不能生成该任务
- `cadence`: `daily | weekly | ad_hoc`
- `maxOccurrencesPerCycle`

### 2.9 `TaskInstance`

当天生成的员工任务。

字段：

- `id`
- `templateId`
- `date`
- `goalId`
- `metricId`
- `title`
- `priority`: `P0 | P1 | P2 | P3`
- `status`: `todo | doing | done | reviewed | skipped`
- `expectedOutcome`
- `actualOutcome`
- `result`: `effective | ineffective | observing | unknown`
- `shouldBecomeSop`
- `notes`
- `evidenceIds`
- `createdBy`: `system | user`
- `blockedByGuardrailIds`

### 2.10 `ExecutionLog`

员工执行记录。

字段：

- `taskId`
- `executedAt`
- `whatWasDone`
- `resultSignal`
- `evidence`
- `evidenceIds`
- `nextAdjustment`

用途：

- 防止软件变成待办工具。
- 建立“动作 -> 结果”的归因基础。

### 2.11 `WeeklyReview`

每周复盘。

字段：

- `weekStart`
- `weekEnd`
- `selectedGoalId`
- `metricSnapshots`: 当周关键指标快照，包含当前值、目标值、分子、分母、风险等级
- `improvedMetricIds`
- `worsenedMetricIds`
- `guardrailViolations`
- `effectiveTaskIds`
- `ineffectiveTaskIds`
- `nextWeekGoalIds`
- `sopEntryIds`
- `summary`
- `nextWeekFocus`: 下周 1-3 个目标
- `openQuestions`: 本周还无法归因的问题

### 2.12 `SopEntry`

SOP 和避坑记录。

字段：

- `id`
- `type`: `effective_sop | pitfall | rule_playbook | method`
- `title`
- `content`
- `relatedMetricIds`
- `relatedRuleSourceIds`
- `createdFromTaskId`
- `lastUsedAt`
- `confidence`: `low | medium | high`
- `status`: `draft | active | retired`
- `evidenceIds`

### 2.13 `MethodCard`

方法论卡片。它不是 SOP，SOP 是店铺实践沉淀；MethodCard 是可复用的方法或官方打法解释。

字段：

- `id`
- `title`
- `type`: `official_rule_playbook | internal_method | checklist`
- `relatedMetricIds`
- `relatedRuleSourceIds`
- `content`
- `steps`
- `warnings`
- `examples`

示例：

- 旺旺 3 分钟响应值班法
- 客服四问法
- 三档报价模板
- 定制订单交付确认清单
- 老客 7 天回访和 30 天补货提醒

### 2.14 `OperatingAsset`

运营资产检查项，用来衡量项目有没有留下可复制能力。

字段：

- `id`
- `type`: `customer_table | sku_table | keyword_table | script | quote_template | delivery_sop | profit_sheet | other`
- `title`
- `status`: `missing | draft | usable | needs_update`
- `lastUpdatedAt`
- `relatedGoalIds`
- `relatedMetricIds`
- `owner`
- `notes`

用途：

- 支撑“资产沉淀数量”指标。
- 防止项目只留下短期订单，不留下客户、SKU、关键词、话术、报价模板、交付 SOP 和利润表。

### 2.15 `Evidence`

任务结果、SOP 和指标录入的证据。

字段：

- `id`
- `type`: `text | screenshot_ref | file_ref | metric_snapshot | link`
- `label`
- `content`
- `createdAt`
- `relatedTaskId`
- `relatedMetricId`
- `relatedSopId`

用途：

- 支持复盘归因。
- 防止员工只勾选完成，没有结果证据。
- 后续可以接入截图识别或后台导入。

## 3. 派生计算

### 3.1 当前目标状态

输入：

- `GoalPreset`
- `MetricReading[]`
- `ShopContext`
- `DecisionGuardrail[]`

输出：

- 是否有资格冲该目标
- 哪些指标不适用于当前店铺
- 哪些指标达标
- 哪些指标未达标
- 最高风险指标
- 是否被经营底线阻断

### 3.2 今日任务生成

输入：

- `DiagnosticGap[]`
- `DecisionGuardrail[]`
- `TaskTemplate[]`
- 昨日任务结果
- 当前 `IterationCycle`

输出：

- 3-5 个今日任务
- 每个任务绑定指标、原因、方法论和结果记录要求

### 3.3 每周迭代判断

输入：

- 本周和上周 `MetricReading`
- 本周 `TaskInstance`
- 本周 `ExecutionLog`
- 本周 `Evidence`

输出：

- 进步指标
- 退步指标
- 可能有效动作
- 无效动作
- 下周目标建议
- SOP 沉淀建议
- 仍无法归因的问题

## 4. 关键数据流

```text
ShopContext + GoalPreset
        |
        v
适用性判断
        |
        v
MetricDefinition + MetricReading
        |
        v
DiagnosticGap
        |
        v
DecisionGuardrail
        |
        v
TaskTemplate -> TaskInstance
        |
        v
ExecutionLog + Evidence
        |
        v
WeeklyReview -> SopEntry / OperatingAsset
```

数据流要求：

- `MetricDefinition` 决定指标是什么，`MetricReading` 只记录某次读数。
- `GoalPreset` 决定目标值，但允许 L 等级类目标手填目标值。
- `DecisionGuardrail` 先过滤危险任务，再生成今日任务。
- `WeeklyReview` 必须保存快照，不能依赖未来可能被修改的读数。

## 5. 第一版存储策略

第一版可以先使用本地 JSON/localStorage，不引入后端。

建议文件/模块拆分：

- `src/types/domain.ts`: 类型定义
- `src/data/ruleSources.ts`: 官方规则来源
- `src/data/metricDefinitions.ts`: 指标定义
- `src/data/goalPresets.ts`: 目标预设
- `src/data/taskTemplates.ts`: 任务模板
- `src/data/methodCards.ts`: 方法论卡片
- `src/lib/diagnostics.ts`: 差距诊断
- `src/lib/guardrails.ts`: 冲突裁判
- `src/lib/taskGeneration.ts`: 任务生成
- `src/lib/weeklyReview.ts`: 周复盘计算
- `src/lib/storage.ts`: 本地存储适配

未来接入数据库时，这些实体可以直接迁移为表。

## 6. 第一版必须测试的模型边界

- 找工厂目标在未入驻找工厂时应被阻断。
- 保温杯消费品店铺优先使用 48 小时揽收率；找工厂冲级同时检查找工厂履约率。
- 低置信度读数不能被当成确定结论。
- 百分比指标缺少分母时应提示补数据。
- P0 风险存在时，应暂停 P2/P3 扩张任务。
- 周复盘要保存指标快照，后续改动历史读数不能改变已生成复盘。
- 任务没有指标绑定时不允许保存为系统任务。
- 任务没有执行证据时，只能标记为 `observing`，不能直接沉淀为高置信 SOP。

## 7. 不建模事项

第一版不建模：

- 真实 1688 账号授权
- 多员工权限
- 订单明细全量同步
- 广告投放结构
- 库存批次
- 财务流水

这些会把系统拉向 ERP，偏离“官方目标驱动的运营迭代驾驶舱”。
